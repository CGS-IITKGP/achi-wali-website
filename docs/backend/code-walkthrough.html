Code walkthrough

A contributor-oriented guide that starts at the API handler and ends at the database and external services.

Canonical request path

  1. app/api/**/route.ts defines the route handler.
  2. createHandler (from lib/handler.ts) executes:
    • Session extraction when requireAuth is true.
    • Body parsing and dataUnifier for path/query data.
    • Zod validation using a schema from lib/validators.
  3. The service function runs (from lib/services).
  4. Services call repositories (from lib/database/repos).
  5. Repositories run Mongoose queries on models in lib/database/models.
  6. ResponseHandler standardizes the JSON response.
Always use createHandler. The older name createServiceOnlyHandler was a legacy typo and should not be used in new code.

Handler structure

createHandler takes a HandlerConfig with a validator schema, auth requirement, and service definition. The handler returns a NextResponse with a uniform payload: { action, data? } on success or { action, message, errors? } on failure. 

createHandler({
  requireAuth: true,
  validationSchema: widgetValidator.create,
  dataUnifier: (req, body) => ({ ...body, slug: getSlug(req) }),
  options: {
    service: widgetService.create,
    onSuccess: (data) => ({ responseData: data }),
  },
});

Route handler example (query + path params)

// app/api/blog/view/[slug]/route.ts
import createHandler from "@/lib/handler";
import blogValidator from "@/lib/validators/blog.validator";
import blogService from "@/lib/services/blog.service";
import { APIControl } from "@/lib/types/api.types";

const GET = createHandler({
  validationSchema: blogValidator.get,
  dataUnifier: (req) => {
    const urlTokens = new URL(req.url).pathname.split("/");
    return {
      target: APIControl.Blog.Get.Target.BY_SLUG,
      slug: urlTokens[urlTokens.length - 1],
    };
  },
  requireAuth: false,
  options: {
    service: blogService.get,
  },
});

export { GET };

Validator example (Zod)

// lib/validators/blog.validator.ts
import { z } from "zod";
import { allIbDField } from "./core.validator";
import { APIControl } from "../types/api.types";

const blogValidator = {
  get: z.object({
    target: z.enum(APIControl.Blog.Get.Target),
    slug: allIbDField.shortString.optional(),
  }).refine((data) => {
    if (data.target === APIControl.Blog.Get.Target.BY_SLUG && !data.slug) {
      return false;
    }
    return true;
  }),
};

export default blogValidator;
All schemas must be Zod. Shared formats belong in core.validator as allIbDField entries.

Validation rules

  • All inputs are validated with Zod.
  • Reusable formats belong in core.validator (see allIbDField).
  • Validators cover both body and query/route params.
// core.validator.ts
const allIbDField = {
  _id: z.string().transform((val) => new Types.ObjectId(val)),
  slug: z.string().regex(/^[a-z0-9]+(?:-[a-z0-9]+)*$/),
  url: z.string().url(),
};

Type system map

Types are the backbone of the backend. Prefer adding new types in lib/types unless the data is truly local to a file.

For the complete reference, see Types catalog.

  • APIControl: enumerates target strings for query-driven handlers.
  • SDIn (ServiceDataIn): input payloads for services.
  • SDOut (ServiceDataOut): successful return shapes for services.
  • ESECs: Service error codes (Enum). The E prefix denotes enums.
  • ServiceSignature: function type for services.
  • IServiceResolve: success/error union returned by services.
  • SuccessResponseCodesEnum, FailedResponseCodeEnum, FailureResponseCodesEnum: HTTP status mapping.
type ServiceSignature =
  (data: SDIn, session: RequireSession extends true ? ISession : ISession | null)
    => Promise>;

Service example (auth + business rules)

// lib/services/project.service.ts (excerpt)
const create: ServiceSignature<
  SDIn.Project.Create,
  SDOut.Project.Create,
  true
> = async (data, session) => {
  if (!session.userRoles.includes(EUserRole.MEMBER)) {
    return {
      success: false,
      errorCode: ESECs.FORBIDDEN,
      errorMessage: "Only members can create a new project.",
    };
  }

  await projectRepository.insert({
    portfolio: data.portfolio,
    title: data.title,
    description: data.description,
    tags: data.tags,
    author: session.userId,
    collaborators: [],
    links: data.links,
    coverImgUrl: data.coverImgUrl,
    media: [],
  });

  return { success: true, data: {} };
};
Use ESECs for expected client errors, and throw AppError for unexpected failures.

Naming conventions

  • Handlers live in app/api/**/route.ts.
  • Services: feature.service.ts.
  • Validators: feature.validator.ts.
  • Repositories: feature.repo.ts.
  • Models: feature.model.ts.
  • Types: feature.types.ts or namespace files in lib/types.

Service layer conventions

  • Services return { success: true, data } or an error code.
  • Authorization lives in services (role checks, ownership checks).
  • Use ESECs for client-facing errors.
  • Throw AppError for unexpected failures.

Repository layer conventions

  • Services always call repositories, never raw models.
  • Generic behavior belongs in GenericRepository.
  • Feature-specific queries live in the feature repository.

Repository example (custom query)

// lib/database/repos/blog.repo.ts (excerpt)
async findAllOfListExportable(filter = {}, session) {
  await this.ensureDbConnection();
  return await this.model
    .find(filter)
    .select("-content")
    .populate({ path: "author", select: "name profileImgUrl" })
    .session(session || null)
    .lean()
    .exec();
}

proxy.ts is the canonical middleware entrypoint for route protection.

Response format

  • Success: { action: true, data: { ... } }
  • Client error: { action: false, message, errors? }
  • Server error: { action: null } with 500 status
